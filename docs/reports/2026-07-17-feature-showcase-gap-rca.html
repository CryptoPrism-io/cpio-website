Section 01

Executive Summary

The live Feature Showcase section has fallen behind its own design source on every one of its six screens. The gap is not cosmetic — it is architectural.

The marketing site's Feature Showcase — the section that carries CryptoPrism's core product pitch to prospects — is currently built from five static PNG screenshots inside a hover-to-expand shelf. The Claude Design source it was originally built from has since moved on to a materially different architecture: six auto-playing tabs, each rendering from a structured, live-feeling mock-data model, with the Analytics tab specifically carrying five real Chart.js visualizations. None of that evolution has made it into the live site.

Critical
Analytics — wrong sub-tab shown, zero charts, no charting dependency installed
Major
Economic Calendar — an entire tab (100% of its content) does not exist locally
Major ×4
Dashboard, AI Screener, News Intelligence, Public Screens — static & partly stale

Three findings matter most for prioritization:

  • The gap compounds, it doesn't average. Every screen carries at least one of three independent problems — stale content, no interactivity, or missing architecture — and Analytics and Economic Calendar carry all three at once.
  • Some of the "static" images are already out of date relative to the current design source (an extra chip on Public Screens, an extra table on Dashboard, extra fields on News Intelligence) — evidence the export-and-forget workflow has already failed once, not just a risk going forward.
  • The fix is not six independent patches. All six screens sit on top of one shared component (FeatureShowcase.tsx) that has no per-screen logic at all — the highest-leverage first move is rebuilding the shared shell once, not touching screens one at a time.
Headline risk
Analytics is the screen most likely to be scrutinized closely by a technically literate prospect or investor (it's the "institutional-grade market depth" claim made visual) — and it is also the screen furthest from the design spec, showing unrelated on-chain metrics instead of the intended market-overview charts, with no charting library even installed to close the gap.
Section 02

Root Cause Analysis

Why six screens, built once from the same source, are now six different distances from that source.

2.1 — The proximate cause: a one-way export, not a living sync

The local implementation renders every screen as a single flat <img>. That image was captured once, at some earlier point in the design's iteration, and then frozen. The design source, by contrast, is live-coded — it kept changing after the export was taken. Three of the six screenshots carry direct evidence of this: Public Screens still shows a "Technical" category chip the current design has since dropped; Dashboard's static image includes an extra Watchlist table and CTA box not present in the current 2-column spec; News Intelligence's image has an extra takeaway, an extra tracked asset, and a "Market Impact" field that doesn't exist in the current data schema. These aren't three unrelated bugs — they're three independent proofs of the same mechanism: there is no process that re-captures the export when the source changes.

2.2 — The structural cause: no per-screen seam to update

FeatureShowcase.tsx (80 lines) has exactly one array of five {key, title, desc, src} objects and one shared render path. There is no per-screen component, no data model, and no dependency on the design's own data structures. This means there was never a cheap way to update "just the Analytics numbers" or "just the Public Screens category chips" — any content update requires either re-exporting a new screenshot (which is what happened, once, and then stopped) or a full rebuild of that screen as real markup (which has not happened at all). The path of least resistance was to do neither, and the section quietly aged in place.

2.3 — The capacity cause: the design source outgrew the build in one leap

The design didn't drift gradually — it re-architected in one revision, adding a sixth tab (Economic Calendar), an autoplay/progress-bar interaction model, and a genuinely new runtime dependency (Chart.js 4.4.3, loaded live in the design tool but never added to package.json). A change of that size does not fit inside "swap the PNG" — it requires a scoped engineering task, and without that task being scheduled, the gap simply sat unaddressed while the design kept moving.

Why this matters for sequencing
Because the root cause is structural (no live seam) rather than incidental (nobody happened to update an image), patching individual screenshots would only reproduce the same failure mode again at the next design revision. Section 05 sequences the fix around rebuilding the shared seam first.

Section 05

Remediation Roadmap

A sequence, not a punch list — because the root cause is structural, the first move has to be the shared shell, not any one screen.
Fig. 2 — Gap Composition by Screen (qualitative analyst scoring, 0–3 scale)
Content drift Interactivity debt Architecture debt

5.1 — Recommended sequence

  1. Stage 0 — Rebuild the shared shell
    Replace the static-image shelf with a data-driven, 6-slot tab component matching the design's pattern: a SCREENS-style array, autoplay with pause-on-hover, and a per-tab progress bar. Add chart.js as a dependency now, since Analytics will need it and it's cheaper to add once. Nothing else should start before this — every screen fix built on the old static-shelf pattern would be thrown away.
  2. Stage 1 — Port the two closest-to-parity screens
    Public Screens and Dashboard have the least content drift — build them as real components first to validate the new shell with the lowest content risk, and pick up their small fixes (drop the stale chip, add the captions/CTA row; reconcile the Dashboard layout to 2 columns) as part of the port rather than as a throwaway patch to the old PNG.
  3. Stage 2 — Rebuild AI Screener and News Intelligence
    Both need real content work, not just a port: AI Screener drops the full-chrome mockup for the lean prompt pane; News Intelligence's data needs reconciling against the current schema (drop the stray takeaway/asset/field, restore the missing headline).
  4. Stage 3 — Build Economic Calendar
    No local precedent to unwind, but the largest net-new surface area of the six (week strip, filters, events table, detail panel with live countdown). Sequenced after the others so the shell and data patterns are already proven.
  5. Stage 4 — Build Analytics
    The most expensive screen: 5 chart configs, an 8-item sub-tab rail, a ticker, and a random-walk mock-data generator. Sequenced last by build complexity — see the tradeoff below before locking this order in.
A sequencing choice worth deciding explicitly
The order above sequences by increasing build complexity, which de-risks the new shell early. An alternative is to sequence by business severity instead — pulling Analytics into Stage 1 right after the shell, since it's the only Critical-rated screen and the one most likely to be scrutinized by a technical prospect, at the cost of taking on its full complexity (new dependency, 5 charts) before the shell pattern has been proven on anything else. Both are reasonable; this report recommends the complexity-first order but flags the alternative for a founder call, not an engineering one.
Section 06

Methodology

How each screen's gap was established — for confidence in the findings above, not for repetition here.

The current Claude Design source (project "Website Cryptoprism," file CryptoPrism Hero.dc.html) was pulled via the DesignSync MCP connector as raw markup and mock-data source, giving exact copy, data values, colors, and structure rather than an approximate screenshot read. Six subagents ran in parallel, one per screen, each given the precise line ranges of its screen's markup and data block plus the shared local component file, and each independently captured a live Playwright screenshot of the corresponding card on the running local site (localhost:5173) to confirm the current rendered state before reporting a gap. Findings across all six were cross-checked for the shared architectural pattern (static image vs. live tab) before being consolidated into this report.
